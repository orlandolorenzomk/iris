== Overview

Iris Nexus is the central control plane of the Iris platform.
It is the *only component exposed to the outside world* — all traffic from Iris Panel (Angular) and Iris CLI flows through it,
and all commands to the internal engines (Prism, Orbit, Aurora) originate from it.

It is built as a *Spring Boot 3 Modulith* and compiled to a *GraalVM native image* to keep idle
resource usage near zero.

== Modules

=== auth

Handles user registration, login, and JWT issuance.
All other modules consume the identity it establishes.

Iris has two roles:

[cols="1,3", options="header"]
|===
| Role    | Description

| `OWNER` | Created once at startup from `.env` values. Has full access to all resources and machines.
           Cannot be deleted — attempting to delete the OWNER account is rejected at the application level.
| `USER`  | Registered through the API. Can only manage their own spaces and deployments.
           No access to machine management or cluster settings.
|===

The `OWNER` account is seeded on first startup if it does not already exist.
Credentials are read from environment variables (`ADMIN_DEFAULT_EMAIL`, `ADMIN_DEFAULT_PASSWORD`).
If the account already exists, the seed step is skipped — subsequent restarts never overwrite it.

The role is embedded in the JWT as a claim and verified by Spring Security on each request.

.Responsibilities
* Seed the OWNER account on first startup from `.env` — skip if already present
* Register new USER accounts via the API with BCrypt-hashed passwords
* Authenticate users and return a signed JWT carrying the role claim
* Expose user identity and role claims to other modules via internal events
* Enforce resource-level ownership (a USER can only access their own spaces and deployments)
* Reject deletion of the OWNER account

=== spaces

Manages the lifecycle of User Spaces and enforces resource quotas.
No deployment can proceed without passing through this module's quota check.

.Responsibilities
* Create and delete User Spaces
* Store per-space quota definitions (CPU, RAM, storage)
* Track current resource usage per space
* Reject deployments that would exceed quota before they reach the deployments module
* Track the network state of each space (`space_networks`) — VNet ID, subnet, VNI, and provisioning status

=== deployments

Orchestrates all deployment requests by coordinating Prism, Aurora, and Orbit.
It does not provision infrastructure directly — it delegates to the correct engine and tracks state.

.Responsibilities
* Consume `DeploymentAllowed` events from the spaces module
* Tell Prism to provision a VM or LXC
* Tell Aurora to resolve the image reference
* Tell Orbit to schedule and run the container
* Persist and expose deployment state

=== machines

Manages the registration, lifecycle, and real-time capacity tracking of Proxmox host machines.
The primary machine — the one Nexus is hosted on — is bootstrapped automatically on first start.
All additional machines are registered manually and connected over SSH.

.Responsibilities
* Auto-configure the primary Proxmox host on startup
* Store connection details for additional machines (IP, SSH port, SSH user)
* Encrypt all credentials at rest using AES secret keys — never exposed through the API
* Bootstrap Prism and Orbit on newly registered remote nodes via SSH
* Receive and persist capacity heartbeats from Prism into `machine_stats`
* Expose node selection logic to the deployments module
* Store cluster-wide infrastructure settings in `cluster_settings` (SDN zone type, zone name, NATS URL) — always a single row
* Store per-node management interface name in `machines.mgmt_iface` — can differ per node (e.g. `eth0` on one node, `bond0` on another)
* Track per-node network apply state in `machine_networks` to drive Prism reconciliation

==== Node Selection

When the `deployments` module needs to place a workload, it calls the `machines` module
to select the best available node.

Selection runs in two steps:

. *Filter* — discard any node whose free CPU, RAM, or disk cannot satisfy the workload requirements
. *Rank* — score each remaining node and pick the highest

The scoring function weighs free capacity as a percentage of total capacity:

[source,java]
----
double score(MachineStats stats) {
    double cpu  = stats.getFreeCpu()  / stats.getTotalCpu();
    double ram  = stats.getFreeRam()  / stats.getTotalRam();
    double disk = stats.getFreeDisk() / stats.getTotalDisk();

    return (cpu * 0.4) + (ram * 0.4) + (disk * 0.2);
}
----

RAM and CPU are weighted equally at 40% each — the primary bottlenecks for most workloads.
Disk is weighted at 20%. If two nodes share the same score, the tiebreaker is the node
with the fewest active deployments.

==== Heartbeat Processing

Every few seconds, each Prism instance publishes its current resource usage to NATS:

----
iris.heartbeat.{node_id}  →  { cpu_free, cpu_total, ram_free, ram_total, disk_free, disk_total }
----

Nexus consumes these events and upserts the `machine_stats` table.
If a heartbeat is missed for more than 30 seconds, the node is marked `unavailable`
and excluded from scheduling until heartbeats resume.

==== Network State Tracking and Reconciliation

Nexus is the source of truth for which Proxmox nodes have each space's VNet applied.
This is tracked in `machine_networks` — a junction between `machines` and `space_networks`.

.`machine_networks.status` values
[cols="1,3", options="header"]
|===
| Status            | Meaning

| `pending_apply`   | Nexus has published the network command to Prism but has not yet received confirmation
| `active`          | Prism confirmed the VNet is live on this node
| `pending_destroy` | Nexus has published the destroy command but Prism has not yet confirmed removal
| `destroyed`       | VNet has been torn down on this node
|===

This table drives two critical operations:

*Space deletion* — when a User Space is deleted, Nexus queries `machine_networks` to find every
node where the space's VNet is `active`, then publishes a destroy command to each node's Prism.
It waits for all entries to reach `destroyed` before marking the space as fully deleted.

.Network teardown across nodes
[mermaid]
....
sequenceDiagram
    actor User
    participant Nexus as Iris Nexus
    participant DB    as PostgreSQL
    participant NATS  as NATS JetStream
    participant P1    as Iris Prism (node-1)
    participant P2    as Iris Prism (node-2)

    User  ->> Nexus: DELETE /spaces/{id}
    Nexus ->> DB:    SELECT machine_networks WHERE space_network_id = X AND status = active
    DB   -->> Nexus: [node-1, node-2]

    Nexus ->> DB:    UPDATE machine_networks SET status = pending_destroy
    Nexus ->> NATS:  iris.prism.node-1.network.destroy {vnet_id}
    Nexus ->> NATS:  iris.prism.node-2.network.destroy {vnet_id}

    P1   ->> NATS:   iris.events.deployments {status: network_destroyed, node: node-1}
    P2   ->> NATS:   iris.events.deployments {status: network_destroyed, node: node-2}

    Nexus ->> DB:    UPDATE machine_networks SET status = destroyed (both rows)
    Nexus ->> DB:    UPDATE space_networks SET status = destroyed
    Nexus ->> DB:    UPDATE spaces SET status = deleted
....

*Node recovery* — when a Prism that was `unavailable` resumes sending heartbeats,
Nexus queries `machine_networks` for all `active` networks on that node and re-publishes
each network command so Prism can re-apply any nftables rules that were lost on reboot.

[NOTE]
====
`cluster_settings` is a singleton table — it always contains exactly one row.
Enforced in PostgreSQL with `lock BOOLEAN DEFAULT true PRIMARY KEY CHECK (lock)`.
The application reads it on startup and caches it; writes are rare (initial setup or admin reconfiguration only).

`machines.mgmt_iface` is stored per node because interface names are not guaranteed to be consistent
across different Proxmox hosts (e.g. `eth0` vs `ens3` vs `bond0`).
Prism uses this value when applying nftables NAT rules and VXLAN peering.
====

== Internal Engine Communication

Nexus communicates with Prism, Orbit, and Aurora asynchronously via *NATS JetStream*
for all management and event-driven operations.

Aurora is a special case: the actual `docker push` / `docker pull` flow must remain HTTP
because the Docker client speaks the OCI distribution spec directly and cannot be changed.
Everything else — image push notifications, deploy triggers from the Panel or CLI, catalog
queries — goes through NATS.

[cols="1,2,2", options="header"]
|===
| Target      | Protocol                     | Reason

| Iris Prism  | NATS JetStream               | Async VM/LXC provisioning across multiple nodes
| Iris Orbit  | NATS JetStream               | Async container lifecycle management
| Iris Aurora | HTTP (OCI) + NATS JetStream  | HTTP for docker push/pull (protocol requirement), NATS for all management and events
|===

[NOTE]
====
User JWTs never leave Nexus. NATS messages carry a service-level credential so Prism,
Orbit, and Aurora reject any message that did not originate from Nexus.
====

== Messaging Architecture

=== Why NATS JetStream

NATS JetStream is the persistence layer on top of NATS core.
It provides at-least-once delivery, consumer acknowledgment, and message replay —
critical for operations like VM provisioning that must not be silently dropped.

The Zig client for Prism and Orbit uses the official *nats.c* C library via Zig's
`@cImport`, requiring zero custom protocol implementation.

=== Subject Naming Convention

All Iris messages follow a structured subject hierarchy:

----
iris.{component}.{node_id}.{action}
----

[cols="2,3", options="header"]
|===
| Subject                          | Description

| `iris.prism.{node_id}.provision` | Provision a VM or LXC on a specific node
| `iris.prism.{node_id}.destroy`   | Destroy a VM or LXC on a specific node
| `iris.prism.{node_id}.network`   | Create or tear down a User Space VNet
| `iris.orbit.{node_id}.schedule`  | Schedule a container task on a specific node
| `iris.orbit.{node_id}.stop`      | Stop a running container task
| `iris.orbit.{node_id}.scale`     | Scale replicas up or down
| `iris.aurora.image.deploy`       | Nexus triggers a deploy using a specific image (Panel or CLI)
| `iris.events.deployments`        | Status events published back to Nexus by Prism/Orbit
| `iris.events.images`             | Aurora notifies Nexus when a new image push completes
|===

The `{node_id}` segment allows Nexus to target a specific Proxmox node.
Each Prism/Orbit instance subscribes only to subjects matching its own node ID.

=== Image Push vs Panel/CLI Deploy

Users have two ways to deploy an image:

[cols="1,2,2", options="header"]
|===
| Method         | Flow                                        | Entry point

| `docker push`  | Docker CLI → HTTP → Aurora → NATS event     | OCI registry HTTP endpoint
| Panel / CLI    | User selects image → Nexus → NATS → Aurora  | Nexus REST API
|===

In both cases, once the image is resolved, Nexus publishes the provisioning commands
to Prism and Orbit via NATS — the rest of the flow is identical.

=== Async Deployment Flow

Operations like VM provisioning are inherently slow (seconds to minutes).
Nexus publishes a command and returns `202 Accepted` immediately.
Prism/Orbit publish status events back on `iris.events.deployments` as work progresses.

[mermaid]
....
sequenceDiagram
    actor Client
    participant Nexus
    participant NATS  as NATS JetStream
    participant Prism as Iris Prism (node-1)
    participant Orbit as Iris Orbit (node-1)

    Client ->> Nexus: POST /deploy
    Note over Nexus: persist deployment (status=pending)

    Nexus  ->> NATS: publish iris.prism.node-1.provision
    Nexus  ->> NATS: publish iris.prism.node-1.network
    Nexus -->> Client: 202 Accepted + deployment_id

    NATS   ->> Prism: iris.prism.node-1.network
    Prism  ->> Prism: create VNet via Proxmox API
    Prism  ->> NATS:  publish iris.events.deployments (network=ready)
    NATS  -->> Prism: ACK

    NATS   ->> Prism: iris.prism.node-1.provision
    Prism  ->> Prism: create VM via Proxmox API
    Prism  ->> NATS:  publish iris.events.deployments (vm=ready, vm_id=...)
    NATS  -->> Prism: ACK

    Nexus  ->> NATS: publish iris.orbit.node-1.schedule (vm_id + image_ref)
    NATS   ->> Orbit: iris.orbit.node-1.schedule
    Orbit  ->> Orbit: pull image + run container
    Orbit  ->> NATS:  publish iris.events.deployments (status=running)
    NATS  -->> Orbit: ACK

    NATS   ->> Nexus: iris.events.deployments (status=running)
    Nexus  ->> Nexus: UPDATE deployment status=running

    Client ->> Nexus: GET /deployments/{id}
    Nexus -->> Client: 200 running
....

=== Stub Implementation During Development

Prism and Orbit do not exist yet. During Nexus development, a stub consumer
runs in-process and immediately acknowledges all commands with a fake success event.

[source,java]
----
@Component
@Profile("stub")
public class StubPrismConsumer {

    @NatsListener(subject = "iris.prism.*.provision")
    public void onProvision(ProvisionCommand cmd) {
        // publish a fake success event back
        eventPublisher.publish(DeploymentEvent.vmReady(cmd.deploymentId(), "stub-vm-" + UUID.randomUUID()));
    }
}
----

This allows the full Nexus request/response cycle — including status polling — to work
end-to-end without any Proxmox infrastructure.

== GraalVM Native Build

Nexus compiles to a native binary using GraalVM ahead-of-time compilation,
eliminating JVM warmup and significantly reducing memory footprint.

[source,bash]
----
./gradlew nativeCompile
----

.Key considerations
* Spring Boot 3.x AOT processing handles most internal reflection automatically
* Custom reflection must be declared via `@RegisterReflectionForBinding` or a `RuntimeHintsRegistrar`
* JJWT 0.12+ ships with native-image metadata — no additional hints needed
* Flyway migrations run at startup and are fully compatible with native images

.Expected native image profile
[cols="1,1,1", options="header"]
|===
| Metric       | JVM     | Native

| Startup time | ~3s     | < 100ms
| Idle RSS     | ~250 MB | ~40 MB
|===
